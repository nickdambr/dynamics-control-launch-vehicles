## Context

HM3 is a frozen-time analysis: plant matrices, PD gains, notch and TVC are
all evaluated at the max-qbar instant (`t_ref = 72 s`) and the closed loop is
LTI. The professor's wind generator (`General/hw3-v3/strong_wind.slx`) instead
spans the whole ascent (0–140 s, altitude- and velocity-scheduled Dryden
turbulence). Today the two are reconciled *upstream*: `load_wind_profile`
(`'profile','strongwind'`) simulates the generator and windows 12 s of wind
around max-qbar — correct for a frozen-point model, but it deliberately
throws away the rest of the trajectory.

The conceptually complete architecture is the opposite one: make the **plant
time-varying** too, so wind generator and vehicle dynamics share the same
clock and the generator can be wired **directly into the Simulink scheme**.
Everything needed already ships with the course data:
`GreensiteLPV_DATA.mat` contains the full time histories `A6(t)`, `K1(t)`,
`a1(t)`, `a3(t)`, `a4(t)`, `V(t)`, `Tc(t)`, `omega(t)` (bending frequency),
`sigma_ins(t)`, `phi_ins(t)`, `phi_tvc(t)`, `h(t)` — the same dataset
`load_hw3_params` already samples at a single instant.

This is a portfolio showcase (gain-scheduled LV attitude control over the
ascent), **not** part of the HM3 deliverable — the assignment explicitly asks
for the max-qbar point design. Do not start before the HM3 report is done.

## Architecture

New model `HM3/models/hm3_full_ascent.slx` (the frozen-time
`hm3_closed_loop.slx` stays as is — it is the validated HM3 deliverable):

1. **LPV plant** — replace the constant State-Space block with elementary
   blocks (integrators + time-varying coefficient gains fed by 1-D lookup
   tables on clock time, breakpoints = `GreensiteLPV.*.Time`). This mirrors
   the style of the professor's own generator (lookups + Varying Transfer
   Function) and keeps every coefficient inspectable. Alternative: the
   Control System Toolbox *LPV System* block gridded on time — less
   didactic, evaluate both.
2. **Wind inside the loop** — copy the generator `Subsystem` block out of
   `strong_wind.slx` into the new model (one-time copy; the professor's file
   is never modified). Total wind `v_wp + turbulence`, converted with the
   **time-varying** `alpha_w = v_w / V(t)` (lookup), drives the plant
   directly. No From Workspace, no windowing.
3. **Controller** — two variants to compare on the same canvas (variant
   subsystem, same pattern as `hm3_closed_loop`):
   - *Frozen*: the max-qbar PD gains held over the whole flight — shows
     where a single-point design degrades;
   - *Gain-scheduled*: `design_controller` run on a grid of frozen plants
     along the trajectory → `Kp_th(t)`, `Kd_th(t)` lookup tables.
4. **Script baseline stays the source of truth** — an LTV ode45 integration
   in MATLAB (`ode_lpv_ascent.m`, coefficients interpolated from the same
   dataset, same wind series logged from the generator) replays the Simulink
   run; overlay tolerance target ~1e-6 on theta, same validation pattern as
   `run_simulink_closed_loop`.

## Implementation plan

1. `init_simulink_lpv.m`: load `GreensiteLPV`, build the coefficient grid,
   design the gain schedule (grid of `design_controller` calls on frozen
   plants every ~5 s), push lookup data to base workspace.
2. Build `hm3_full_ascent.slx` per the architecture above (rigid LPV plant
   first; bending/notch is a stretch goal because `omega(t)` would need a
   *Varying Notch* — see Notes).
3. `main_full_ascent.m`: run frozen vs scheduled gains, produce:
   - `theta`, `z`, `delta` over 0–140 s, both controllers;
   - the structural-load indicator `qbar(t)*alpha_total(t)` with the max-q
     region highlighted (the quantity HM3's point design actually protects);
   - frozen-time margin sweep: GM/PM of the loop frozen at each grid point,
     plotted vs flight time (quantifies how far the max-q design stretches).
4. Validation: LTV ode45 replay vs Simulink overlay figure, plus a
   consistency check that at `t = t_ref` the LPV loop matches the HM3
   frozen-time response (same wind window → same traces to ~1e-6).
5. Short README section in `HM3/` ("Beyond the assignment: full-ascent LPV
   simulation") with the comparison figures.

## Notes

- **Do not modify `strong_wind.slx`** — copy its generator subsystem once
  into the new model and note the provenance in an annotation.
- The drift states (`z`, `zdot`) are perturbations normal to the reference
  trajectory; over 140 s their interpretation weakens — state this limit in
  the README section rather than over-engineering the kinematics.
- Bending stretch goal: `omega(t)` spans the lookup range, so the fixed
  min-phase notch of HM3 detunes; the natural tool is the *Varying Notch
  Filter* block (cstextras, same family the professor uses for the Dryden
  filter). Without it, keep the rigid LPV plant — the showcase is the
  scheduling, not the flex model.
- Gain-schedule design reuses `design_controller` unchanged (one frozen
  plant per grid point) — no new tuning machinery.
- Related: T-trade documented in `HM3/models/SIMULINK_GUIDE.md` appendix
  (windowed wind for the frozen plant vs integrated wind for an LPV plant);
  HM3 README "guided Simulink track" section.
- Dependencies: Control System Toolbox (already required); cstextras blocks
  only for the stretch goals.
